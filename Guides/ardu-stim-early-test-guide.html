Ardu-Stim Setup for Early ME7 Testing

This is the practical way to use Ardu-Stim as an early validation tool. It helps you test crank/cam sync behavior first, before full IRONLOOP closed-loop hardware is complete.

Ardu-Stim is a temporary bring-up tool for early phases. It does NOT replace your Pi bridge + engine-sim + full loop path.
What Ardu-Stim Helps With
Scope
Question Ardu-Stim Answer Status
Can ME7 detect crank/cam timing and show RPM? Yes. This is exactly where Ardu-Stim is strong. Yes
Can it replace full engine-sim closed loop? No. It is trigger generation only, not your whole simulation feedback loop. No
Can it reduce Phase 3 risk? Yes. You can prove signal path and ECU sync before writing final Pico logic. Yes

Software + Firmware Setup
First Boot
  1. 1) Install Ardu-Stim GUI Download latest release from: speeduino/Ardu-Stim releases.
  2. 2) Connect Nano by USB Confirm COM port appears in Device Manager.
  3. 3) Upload firmware from GUI First-time Nano setup requires firmware upload. Wait for successful flash message.
  4. 4) Select trigger pattern and RPM mode Start with low RPM and stable fixed mode first. Do not jump straight to high speed.
  5. 5) Validate output with logic analyzer Confirm expected pulse pattern, duty, and sync behavior before ECU connection.

Arduino Nano / Uno D8 D9 D10 A0

These are logic-level outputs. Your ME7 input path may require conditioning or level adaptation depending on whether your target path behaves like Hall or VR-style input expectations.

Recommended Early Test Sequence
Do This Order
  1. Step A: Ardu-Stim bench only Generate signals and verify on logic analyzer. No ECU yet.
  2. Step B: ECU powered safely, no trigger input yet Confirm power rails, fusing, and stable current.
  3. Step C: Connect conditioned crank only Check for RPM detection behavior.
  4. Step D: Add cam signal Verify sync behavior and any ECU fault-code changes.
  5. Step E: Log and compare Capture what works, freeze settings, then move to your Pi/Pico final path.

What This Saves You

  • Reduces unknowns before complex full-loop integration
  • Lets you separate "signal problem" from "software loop problem"
  • Gives known-good baseline trigger behavior for later Pico migration

What It Does Not Replace

  • Pi bridge conversions (TPS/MAP/O2/ECT/IAT)
  • Full engine-sim state-driven closed loop
  • Final production-like crank/cam integration architecture

Current Best Use for You

  • Use Ardu-Stim as temporary trigger test hardware
  • Keep building IRONLOOP around engine-sim + Python bridge
  • Switch to Pico once trigger behavior is validated
