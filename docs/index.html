Overview

The Concept

The goal is to close the loop between a virtual combustion engine running in ange-yaghi/engine-sim and a real Bosch ME7.5 ECU sitting on a bench. The ECU receives electrically-accurate fake sensor signals generated from the simulation state. The ECU's outputs â€” injector timing, ignition advance, idle valve position â€” are read back and fed into the simulation, affecting engine behavior. This is what automotive engineers call a Hardware-in-the-Loop (HIL) rig. Professional versions cost $100k+. This version costs under $200.

In practice, this is a bench-safe tuning practice tool: run trial-and-error against a virtual engine loop first, then carry what works to a real car.

engine-sim â€” Virtual Engine

C++ simulation runs in real time. Outputs RPM, throttle %, manifold pressure, temperatures, lambda, and cam/crank timing. Patched to broadcast state via local socket every frame.

Python Bridge â€” Pi 5

Receives engine-sim state JSON. Converts each value to the correct real-world signal type. Dispatches analog voltages via DAC, resistance values via digital pot, and sends crank/cam timing to the Pico.

Signal Boards â€” DAC + Digital Pot

MCP4728 quad DAC generates real 0â€“5V analog voltages for TPS, MAP, MAF, O2. MCP4131 digital potentiometer fakes NTC thermistor curves for coolant and intake temp sensors.

Pico â€” Crank / Cam Signal Generator

Dedicated Raspberry Pi Pico uses PIO (Programmable I/O) state machines to generate a perfectly-timed 60-2 crank wheel signal and cam sync pulse at the exact frequency matching engine-sim's current RPM output.

ME7.5 ECU â€” Real Hardware

Receives all signals, believes it is running a live engine. Calculates fueling, ignition timing, boost targets, idle control. Fires real injector and ignition driver outputs.

Read-Back â†’ Close the Loop

Python reads ECU outputs (injector pulse width, ignition advance angle, idle valve duty) via GPIO or K-Line. These feed back into engine-sim â€” e.g. more ignition advance = more torque = RPM climbs. Loop repeats at ~60Hz.

Every signal the ME7.5 expects must be electrically generated. Signals vary in type and timing precision required â€” analog voltage, resistance, and hard real-time digital pulses.

Hard Real-Time Digital

60 tooth wheel with 2 missing teeth. Sub-millisecond timing at all RPM. This is the hardest signal. Handled by Pico PIO only â€” not soft PWM.

Hard

The crank signal is the one thing that cannot tolerate jitter. Any timing error in the 60-2 pattern causes the ME7 to lose sync, throw a crank sensor fault, and cut fuel. The Raspberry Pi Pico's PIO subsystem is the key tool here â€” it runs independently of the ARM core at hardware speed and can generate the pattern with microsecond precision regardless of what Python is doing on the main CPU.

pico_crank.py â€” PIO-based 60-2 generator concept 
import rp2, machine, utime
from machine import UART

# PIO state machine for 60-2 tooth pattern
@rp2.asm_pio(sideset_init=rp2.PIO.OUT_LOW)
def crank_pio():
    wrap_target()
    set(pins, 1).side(1)[15]   # tooth high
    set(pins, 0).side(0)[15]   # gap low
    wrap()

uart = UART(0, baudrate=115200)
sm = rp2.StateMachine(0, crank_pio, freq=125_000_000, sideset_base=machine.Pin(0))
sm.active(1)

while True:
    line = uart.readline()
    if line and line.startswith(b"RPM:"):
        rpm = int(line[4:].strip())
        # recalculate PIO clock divider for new RPM target
        tooth_hz = (rpm / 60) * 58  # 58 real teeth on 60-2
        sm.active(0)
        sm = rp2.StateMachine(0, crank_pio, freq=int(tooth_hz * 32), sideset_base=machine.Pin(0))
        sm.active(1)

AI Tuning Layer
Reinforcement learning, auto map optimization
Ongoing

With a stable closed loop, plug in an AI layer on top. The simulation can run far faster than real time once the signal hardware is decoupled, meaning thousands of virtual dyno pulls can be run overnight to evolve optimal fuel and ignition maps.

  • Log full simulation runs to CSV: RPM, load, throttle, lambda, torque, injector PW, advance
  • Build a Python RL environment (OpenAI Gym-style) wrapping the HIL loop
  • Define reward function: max torque at each RPM/load cell, penalize knock risk (over-advance) and rich/lean excursions
  • Train agent (start with PPO via stable-baselines3) to adjust fuel and ignition map cells
  • Export learned maps back to ME7 ROM format for real-world flashing via MPPS or Galletto
  • Validate AI-generated map on real engine (carefully, on a proper dyno)

AI Applications

The AI Layer

Once the HIL loop is closed and stable, AI stops being a gimmick and starts being genuinely useful. The simulation can run headlessly at 10â€“50x real time, meaning an AI agent can explore the ECU map search space rapidly.

Reinforcement Learning Tuning

RL agent treats each map cell as an action. Reward function = torque output minus penalties for knock risk and emissions. Runs thousands of virtual dyno pulls to find optimal values.

Anomaly Detection

Train a model on "healthy" HIL loop data. Deploy it to flag unusual ECU behavior â€” unexpected fueling corrections, erratic advance behavior â€” potentially catching faults before the ECU does.

Map Discovery in ROM Dumps

Feed raw ME7 ROM binary into a pattern recognition model to auto-identify fuel maps, ignition tables, boost targets â€” useful for unknown or poorly-documented ECU variants.

Predictive Sensor Modeling

Train surrogate models on logged HIL data so the simulation can predict what a real engine would do without running the full hardware loop â€” useful for rapid map screening.
