Project IRONLOOP

Engine State Socket Output

Phase 01 Research Log - March 5, 2026

Why This Matters

Phase 01 is blocking the rest of IRONLOOP. If engine-sim cannot publish state in real time, the Python bridge cannot generate sensor outputs for the ME7 bench. No state stream means no closed loop in later phases.

What We Learned So Far

Build/Platform Reality

  • The fork now builds successfully in build-vs18-md with VS 2026 toolchain + NMake.
  • CMake 3.28 does not directly target VS 2026 generator, so VsDevCmd.bat workflow is required.
  • Runtime includes SDL2/SDL2_image and legacy DirectX components.

Compatibility Repairs Already Applied

  • Removed stale assert in piston simulator destructor.
  • Updated legacy Boost filesystem call in piranha path handling.
  • Removed forced debug D3DX link dependencies in delta-studio CMake.

Socket Output Purpose

  1. Provide live engine telemetry to Python bridge.
  2. Feed signal conversion layer (TPS, MAP, MAF, temps, lambda).
  3. Support logging/validation and future closed-loop control.

Minimum State Payload

  • timestamp_ms, frame_number
  • engine.rpm
  • engine.throttle_position
  • engine.manifold_pressure
  • engine.coolant_temp
  • engine.intake_air_temp
  • engine.lambda

Follow-on values: injector pulse width, ignition advance, cam/crank angle, fuel flow.

Transport Decision

Start with UDP on 127.0.0.1:5555 and JSON per simulation frame. It is simple, fast, and good enough for telemetry where occasional packet loss is acceptable.

Implementation Plan

  1. Add a small state broadcaster module in engine-sim (Winsock2).
  2. Hook one publish call into per-frame simulation flow after state update.
  3. Serialize to stable JSON schema.
  4. Add Python receiver in python-bridge to verify schema/rate.
  5. Measure frame impact and fail safely if socket errors occur.

Execution Goal for Next Session

Implement MVP telemetry output with just RPM + throttle + timestamp, verify receiver on localhost, then expand payload field-by-field.
